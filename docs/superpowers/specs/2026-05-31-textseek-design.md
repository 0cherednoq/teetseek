# TextSeek — дизайн-документ

**Дата:** 2026-05-31
**Статус:** утверждён к реализации
**Источник требований:** `docs/textseek_requirements.md`

## Цель

`textseek` — небольшая Python-библиотека без внешних зависимостей для поиска и
извлечения структурированных данных (коды подтверждения, ссылки, произвольные
regex-совпадения) из обычного текста. Библиотека не привязана к источнику текста
(email, SMS, HTML, API, логи и т.д.) — источник остаётся внешним слоем.

Полная спецификация поведения и публичного API — в `docs/textseek_requirements.md`.
Этот документ фиксирует **архитектурные и поведенческие решения**, которые ТЗ
оставляет открытыми.

## Решения, принятые на брейншторме

- **Объём первого прохода:** всё сразу — ядро + полный набор тестов + README +
  полная документация Mintlify.
- **Стратегия поиска кодов:** выбирается эмпирически по корпусу реалистичных
  моков (письма регистрации / 2FA с кодами и ссылками). Внешнее поведение
  фиксировано (см. ниже); конкретная внутренняя реализация выбирается по тестам.
- **Тестирование:** TDD на `pytest`.

### Упаковка

- Сборка через `hatchling`; `pyproject.toml` обновляется под пакет `textseek`.
- Заглушка `main.py` удаляется.
- Dev-зависимость: `pytest`.
- Рантайм-зависимостей — **ноль**. Используется только stdlib: `re`,
  `urllib.parse`, `dataclasses`, `typing`.

### Диспетчеризация

`TextSeek` хранит реестр `тип_спеки → экстрактор`. Каждый экстрактор реализует
`Protocol`:

```python
class Extractor(Protocol):
    def find_all(self, text: str) -> list[ExtractResult]: ...
```

Методы `TextSeek` — тонкий слой поверх экстракторов:

| Метод | Поведение |
|---|---|
| `extract_all(spec)` | весь список совпадений (пустой, если нет) |
| `extract_first(spec)` | первое совпадение или `None` |
| `extract_one(spec)` | первое совпадение или `ExtractNotFoundError` |
| `contains(spec)` | `bool` |

Вся логика поиска изолирована в экстракторах; `TextSeek` отвечает только за
диспетчеризацию и перегрузки типов.

## Поиск кодов

Из включённых типов символов строится класс символов и ищутся **максимальные
«токены»** этого класса нужной длины, ограниченные lookaround'ами.

### Состав класса символов

- `digits=True` → `0-9`
- `alphabet=True` → буквы; уточняется `uppercase`/`lowercase`
  (если `alphabet=True`, но обе границы регистра `False` → `InvalidCodeSpecError`)
- `symbols=True` → набор по умолчанию `-` и `_`

Набор `symbols` по умолчанию — `{'-', '_'}`. Этого достаточно для примеров ТЗ
(`A8F3-K2P9`, `abC_123-xY`); точка/запятая остаются разделителями, поэтому код в
конце предложения корректно обрезается. Расширение набора — задача будущих версий.

### Строгие границы

Кандидат — это **цельный токен**, ограниченный символами не из класса. Следствие:

```python
TextSeek("1234567890").extract_first(Extract.code(length=6, digits=True))  # -> None
```

`123456` внутри `1234567890` не считается кодом. Реализуется через negative
lookbehind/lookahead на символы класса.

### Длина

- `length` задаёт точную длину.
- `min_length`/`max_length` задают диапазон.
- `length` нельзя комбинировать с `min_length`/`max_length`; `min_length > max_length`
  → `InvalidCodeSpecError`.

### `require_each_type`

Пост-фильтр на кандидате: если включено, токен обязан содержать хотя бы один
символ каждого включённого типа (см. раздел 8 ТЗ).

### `near`

1. Найти фразу `near` (регистронезависимо, как подстроку).
2. Взять окно `±window` символов вокруг найденной фразы.
3. Среди кандидатов в окне выбрать **ближайший к фразе**.

Смещения `start`/`end` в результате — в координатах исходного текста.

### `pattern`

Если задан — используется напрямую вместо сгенерированного класса; остальные
параметры состава символов игнорируются (валидируется как корректный regex).

### Эмпирический выбор реализации

Внутренняя реализация поиска кандидатов (явная токенизация vs единый regex-скан)
выбирается по прогону на корпусе моков. Внешнее поведение в обоих вариантах
одинаковое и зафиксировано выше.

## Ссылки

1. Поиск URL-кандидатов regex'ом (`https?://…` до пробела/конца строки).
2. Обрезка хвостовой пунктуации (`.,;:)]}>"'`).
3. Разбор через `urllib.parse.urlsplit` + `parse_qs`.
4. Сборка `LinkResult` (модель из раздела 10 ТЗ дословно: `value/start/end/scheme/
   domain/host/port/path/query/fragment/query_params`).

### Фильтры `Extract.link`

| Параметр | Логика |
|---|---|
| `scheme` | точное совпадение |
| `domain` | точное совпадение host; при `allow_subdomains=True` — также `*.domain` |
| `path` | точное совпадение пути |
| `path_contains` | подстрока в пути |
| `required_query` | наличие ключей (значения не сравниваются) |
| `sample` | парсится в набор фильтров `scheme + domain + path + required_query(ключи)` |

`sample` с некорректным URL → `InvalidSampleLinkError`. Поиск по примеру
сравнивает только структуру и наличие query-ключей, но не их значения
(раздел 11.5 ТЗ).

## Regex

- Компиляция паттерна; битый паттерн → `RegexExtractError`.
- `finditer` по тексту.
- `value` = группа 1, если в паттерне есть группы, иначе полное совпадение
  (раздел 13 ТЗ).
- `RegexResult`: `value/start/end/groups/groupdict`.

## Результаты

Все результаты — `@dataclass(slots=True, frozen=True)` с обязательными полями
`value: str`, `start: int`, `end: int` (раздел 14 ТЗ). Конкретные модели —
`CodeResult`, `LinkResult`, `RegexResult` по разделам 10, 13, 14.

## Ошибки

Иерархия из раздела 17 целиком:

```
TextSeekError
├── ExtractNotFoundError
├── InvalidExtractSpecError
├── InvalidSampleLinkError
├── InvalidCodeSpecError
└── RegexExtractError
```

Валидация спеков — в момент создания через `Extract.code`/`Extract.link`/
`Extract.regex` (а не при поиске). Примеры триггеров `InvalidCodeSpecError`:
ни один тип символов не включён; `length` вместе с `min/max`; `min_length >
max_length`; `alphabet=True` при `uppercase=False, lowercase=False`.

## Типизация

- `@overload` на `extract_one`/`extract_first`/`extract_all` по каждому типу спеки
  (раздел 16 ТЗ), чтобы IDE выводила конкретный тип результата.
- `Literal` где уместно; без `Any` в публичном API без необходимости.
- Аннотации типов во всех публичных методах.

## Тестирование (TDD, pytest)

- Тест на каждый публичный метод и каждый сценарий ТЗ.
- Корпус из ~10–15 реалистичных писем как fixtures: подтверждение регистрации,
  2FA, magic-link, письма с лишними числами/ссылками-«ловушками». Корпус служит
  основой эмпирического выбора стратегии поиска кодов.
- Тесты на валидацию спеков (все ветки ошибок).
- Тесты на строгие границы кодов, `near`, `require_each_type`, фильтры ссылок,
  поиск по `sample`, regex с группами и без.
